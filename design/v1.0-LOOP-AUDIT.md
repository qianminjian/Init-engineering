## 第一部分：三个参考框架分析

### 1.1 LangGraph — 吸取与回避

**核心源码阅读**：`pregel/_loop.py`(PregelLoop)、`graph/state.py`(StateGraph)、`pregel/_algo.py`(prepare_next_tasks)、`checkpoint-sqlite/`(SQLite 持久化)

**应吸取的最佳实践**：

| # | 模式 | 源码位置 | 本项目落地方式 |
|---|------|---------|--------------|
| 1 | **tick/after_tick 分离** — tick() 只决定"下一步做什么"并检查中断，after_tick() 负责持久化和状态应用 | `_loop.py:592-712` | 设计中已有，确认正确 |
| 2 | **prepare_next_tasks 的 PULL+PUSH 双模式** — PULL 从图的边推导，PUSH 从 Send() 动态注入 | `_algo.py:392-468` | 设计中已有 PUSH 预留，v1.0 只需 PULL |
| 3 | **checkpoint 三元组** — `(checkpoint_id, parent_checkpoint_id, checkpoint JSON)` 构成版本链 | `checkpoint/base.py` | **设计中缺失**：需补充 checkpoint schema |
| 4 | **StateGraph builder 链式 API** — `add_node().add_edge().add_conditional_edges().compile()` 返回 Self | `state.py:915-1017` | 设计中已有，确认正确 |
| 5 | **should_interrupt() 检查** — 在 execute 前后检查当前任务名是否在 interrupt_before/after 集合中 | `_loop.py:660-712` | 设计中已有，确认正确 |
| 6 | **retry_policy 机制** — 每个 node 可配置重试策略，`run_with_retry()` 统一处理 | `_runner.py` | **设计中缺失**：应加 RetryPolicy |

**应当回避的复杂度**：

- Channel 类型系统（LastValue/Topic/EphemeralValue/DeltaChannel）— 我们只需单一 state 对象
- Superstep 概念 — 我们单任务串行，不需要批量调度
- FuturesDict/PregelRunner 并行执行 — v1.0 同步即可
- Subgraph 嵌套图 — v2.0+
- ManagedValue/ManagedValueSpec — 不适用
- Durability 多模式（exit/sync/async）— 我们只需 sync

**关键发现**：LangGraph 的核心洞察是 **"控制流在图编译时确定，数据流在运行时通过 channel 传递"**。但它的复杂度大部分来自对通用性的追求（任意 channel 类型、任意 reducer、多 durability 模式）。我们的设计正确地去掉了这些。

### 1.2 AutoGen — 吸取与回避

**核心源码阅读**：`_single_threaded_agent_runtime.py`(1030行完整阅读)、`_intervention.py`、`_base_agent.py`

**应吸取的最佳实践**：

| # | 模式 | 源码位置 | 本项目落地方式 |
|---|------|---------|--------------|
| 1 | **InterventionHandler Protocol** — `on_send/on_publish/on_response` 三个钩子，DefaultInterventionHandler 提供默认透传 | `_intervention.py:20-83` | 设计中已有 on_execute_start/on_execute_end，但应加 **DropMessage 显式丢弃信号** |
| 2 | **延迟 Agent 实例化** — `_agent_factories` dict 存工厂函数，`_get_agent()` 首次访问时才实例化 | `_single_threaded_agent_runtime.py:259-261, 976-986` | 设计中已有 register + _get_or_create_agent，确认正确 |
| 3 | **消息队列驱动循环** — `_process_next()` 从 Queue 取消息 → 匹配处理 → 递归循环 | `_single_threaded_agent_runtime.py:671-794` | v1.0 不需要消息队列，直接同步调用即可。但循环结构可参考 |
| 4 | **AgentType 值对象** — 简单的 `@dataclass(eq=True, frozen=True)` 包装字符串 | `_agent_type.py` | 可选：Stage.name 当前是 str，可以保持简单 |
| 5 | **register_factory 类型检查** — 注册时检查 expected_class，防止类型错误 | `_single_threaded_agent_runtime.py:886-914` | **设计中缺失**：应加 Agent 注册时的类型验证 |

**应当回避的复杂度**：

- Pub/Sub 主题系统（TopicId/SubscriptionManager/TypeSubscription/TypePrefixSubscription）— v1.0 Agent 直接调用，不需要发布订阅
- SerializationRegistry — 不需要自定义序列化
- AgentId with namespace/key — 我们只需简单的 agent_type 字符串
- send_message vs publish_message 区分 — 我们只有直接的 Agent.execute()
- asyncio.Queue 消息传递 — v1.0 同步调用即可
- CancellationToken — 我们不需要取消语义
- OpenTelemetry 追踪 — 后续迭代
- RoutedAgent + @message_handler 装饰器 — 太魔法
- AgentInstantiationContext context vars — 隐式上下文传递不利于测试

**关键发现**：AutoGen 的 InterventionHandler 是中间件模式的典范实现。我们的设计采用了它的思路（on_execute_start/on_execute_end），但**缺少 DropMessage 模式**——建议补充。另外 AutoGen 的 pub/sub 系统是它最复杂的部分，本项目完全不需要。

import json, re

def extract_json_from_response(text: str) -> dict:
    """从 LLM 文本响应中提取 JSON 块。参考 CrewAI converter.py 的 _JSON_PATTERN。"""
    match = re.search(r'\{.*\}', text, re.DOTALL)

### 3.8 BaseAgent 优化版（修复 P1-#7, #8, #9）

```python
# agents/base.py
from dataclasses import dataclass
from auto_engineering.llm.anthropic_provider import AnthropicProvider, LLMResponse
from auto_engineering.agents.parser import extract_json_from_response

@dataclass
class RetryConfig:
    max_attempts: int = 3
    initial_delay: float = 1.0
    backoff_factor: float = 2.0
    timeout: float = 120.0

class BaseAgent:
    role: str
    system_prompt: str
    tools: list[BaseTool]
    llm: AnthropicProvider
    retry_config: RetryConfig = field(default_factory=RetryConfig)
    output_schema: dict | None = None  # 期望的输出 JSON schema

    async def execute(self, task: Task, ctx: TaskContext) -> TaskResult:
        messages = self._build_messages(task, ctx)

        for attempt in range(self.retry_config.max_attempts):
            try:
                response = await asyncio.wait_for(
                    self._call_llm_loop(messages),
                    timeout=self.retry_config.timeout,
                )
                values = self._parse_output(response.text, task)
                return TaskResult(
                    agent_type=self.role,
                    values=values,
                    raw_response=response.text,
                    tool_calls=response.tool_calls,
                )
            except asyncio.TimeoutError:
                if attempt == self.retry_config.max_attempts - 1:
                    raise
                await asyncio.sleep(
                    self.retry_config.initial_delay * (self.retry_config.backoff_factor ** attempt)
                )
```

### 5.2 抄写纪律

1. **先读后写**：打开参考文件 → Read 完整上下文 → 理解为什么要这样写 → 关闭参考 → 按本设计写
2. **只抄模式，不抄实现**：比如抄 LangGraph 的"tick 返回 bool"模式，不抄它的 superstep/channel 系统
3. **本设计是唯一真相源**：如果参考代码的做法和本设计不同，以本设计为准
4. **简化是美德**：参考代码可能因为通用性有大量分支。我们只需要覆盖 v1.0 场景的分支
5. **写完即验**：每个文件写完跑 `python -c "from auto_engineering.xxx import YYY"` 确认无 import 错误

### 5.3 实施顺序

Phase 1（核心引擎）遵循依赖链：
```
state.py → checkpoint.py → graph.py → loop.py
```
理由：loop 依赖 graph，graph 依赖 state，checkpoint 存储 state。
messages.py（Send）在 graph.py 之后，因为条件边会触发 Send。

Phase 1 验收标准：
- `LoopEngine(stage_graph).run("写一个 hello world")` 可在 mock runtime 下跑通 tick 循环
- `checkpoint.save()` 写入 SQLite，`checkpoint.load()` 恢复状态
- 覆盖 `architect → developer → critic → (MAJOR → developer → critic) → APPROVE → END` 全路径

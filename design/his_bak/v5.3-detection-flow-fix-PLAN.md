# v5.3 检测信息流修复 — 设计计划

> 创建：2026-07-14 | 来源：ae-init 存量项目严重缺陷报告
> 基于 v5.0 设计文档的原始意图，修复 detect → prompt → render 信息链路断裂

---

## 1. 根因诊断

### 当前信息流（断裂）

```
┌─────────────────────────────────────────────────────────────────┐
│ Phase 1: detect                                                 │
│   analyze_java() 提取:                                          │
│     java_version=8  spring_boot_version=2.3.x                   │
│     packaging=pom   is_multi_module=True                        │
│     group_id=xxx    artifact_id=yyy                             │
│                          ↓                                      │
│   DetectionResult._java_info ← 存储了所有信息                    │
│   DetectionResult.as_answers() ← detected_java_version,         │
│     detected_spring_boot_version, project_type, project_name    │
│                          ↓                                      │
│ Phase 2: prompt                    ← ⚠ 信息断裂点               │
│   phase_prompt() L95-98:                                        │
│     if k in ("language","package_manager","test_runner",        │
│              "ci_platform","use_lefthook","use_docker"):        │
│         answers.defaults[k] = v                                 │
│                          ↓                                      │
│   ✗ detected_java_version     → 被过滤掉                        │
│   ✗ detected_spring_boot_version → 被过滤掉                     │
│   ✗ project_type              → 被过滤掉                        │
│   ✗ project_name              → 被过滤掉                        │
│                          ↓                                      │
│ Phase 3: render                                                 │
│   Jinja2 context: {{ java_version }} = "21" ← 模板默认值         │
│   实际项目: Java 8                                               │
│   结果: 生成 Java 21 pom.xml → 完全不可用                        │
└─────────────────────────────────────────────────────────────────┘
```

### 设计的原始意图（v5.0 §1）

> "存量项目：通过代码分析自动识别项目类型、依赖、配置"  
> "自动检测, 只增不改, 不修改 src/ 任何已有文件"

这意味着：
- 检测到的信息**必须**驱动模板渲染
- 存量项目检测到的配置**必须**覆盖模板默认值
- 增量模式**必须**感知已有文件结构，而非机械"跳过已存在"

### 缺失的能力

| 设计承诺 | 实际情况 |
|---------|---------|
| 自动识别依赖、配置 | 识别了但被过滤掉 |
| 检测结果驱动渲染 | 检测结果在 prompt 阶段丢弃 |
| 只增不改 | 全量生成 pom.xml 等核心文件 |
| 多模块项目适配 | 按单模块模板生成根 src/ |

## 2. 修复方案

### 2.1 核心变更：打通 detect → prompt → render 信息链路

**phase_prompt() 不再过滤 detection.as_answers() 的字段。**

```python
# 之前（phases/prompt.py:91-98）— 白名单过滤
if detection is not None:
    for k, v in detection.as_answers().items():
        if k in ("language", "package_manager", "test_runner",
                 "ci_platform", "use_lefthook", "use_docker"):
            answers.defaults[k] = v

# 之后 — 全量通过，detection 结果优先级高于模板默认值
if detection is not None:
    for k, v in detection.as_answers().items():
        answers.defaults[k] = v  # 所有检测结果写入 defaults 层
```

**效果**：`detected_java_version` 进入 AnswersMap → 模板中 `{{ detected_java_version }}` 可用。

### 2.2 模板变量映射：detected_* → 模板变量

在 `DetectionResult.as_answers()` 中增加直接变量名映射：

```python
def as_answers(self) -> dict[str, object]:
    result = {}
    # ... 现有逻辑 ...
    if self._java_info:
        for k in ("java_version", "spring_boot_version", 
                  "kotlin_version", "scala_version", "groovy_version"):
            v = self._java_info.get(k)
            if v:
                result[f"detected_{k}"] = v
        # 新增: 直接映射模板变量名（存量项目用）
        if self._java_info.get("packaging"):
            result["java_packaging"] = self._java_info["packaging"]
        if self._java_info.get("is_multi_module"):
            result["is_multi_module"] = True
        if self._java_info.get("group_id"):
            result["java_group_id"] = self._java_info["group_id"]
        if self._java_info.get("artifact_id"):
            result["java_artifact_id"] = self._java_info["artifact_id"]
```

### 2.3 模板修改：优先使用检测值

```jinja
{# templates/monorepo/java/pom.xml.jinja #}
<java.version>{{ detected_java_version or java_version }}</java.version>
```

```jinja
{# templates/_shared/CLAUDE.md.jinja #}
- 测试命令: {{ test_runner or 'npm test' }}
- 构建命令: {{ package_manager or 'npm' }} {{ 'test' if test_runner == 'junit' else 'run build' }}
```

### 2.4 增量模式增强：多模块感知

```python
# Phase 1 detect: analyze() 检测到 is_multi_module=True 时
# → DetectionResult 标记
# Phase 3 render: build_template_dirs() 收到 context 中的 is_multi_module
# → 跳过根级 src/ 模板
# Phase 5 finalize: merge_incremental 已有"跳过已存在"逻辑，无需改动
```

### 2.5 存量项目 src/ 保护

在 `scaffold_render.py:build_template_dirs()` 中：

```python
# 存量增量模式 + 多模块项目 → 跳过 src/ 目录模板
if context.get("is_multi_module") or context.get("_mode") == "incremental":
    exclude.append("src/**")  # 不生成根级 src/
```

### 2.6 Agent 注入点：external_data 桥

设计文档 §5.2 已定义 `_LazyExternalDict`。存量项目场景中，Agent 可以：

1. 深入扫描 pom.xml 提取模块列表、依赖版本、入口类
2. 将分析结果写入 YAML 文件
3. 在 ae-template.yml 中配置 `_external_data` 引用
4. 模板中通过 `_external_data.module_list` 消费

无需修改 init 核心代码——这已是设计的一部分，只是从未被使用。

## 3. 变更范围

### 本次修复（必须）

| 文件 | 变更 | 风险 |
|------|------|------|
| `phases/prompt.py:91-98` | 移除白名单过滤，全量通过 detection.as_answers() | 低：仅增加 defaults 层键，不改变优先级 |
| `detector_constants.py:as_answers()` | 增加 java_packaging/is_multi_module/java_group_id/java_artifact_id | 低：纯数据映射 |
| `phases/prompt.py:95-98` | detection.as_answers() 的 project_name 覆盖模板默认（原注释"不覆盖"修改） | 中：存量项目的 project_name 应从 artifact_id 提取 |
| `scaffold_render.py:build_template_dirs()` | 增量模式 is_multi_module 跳过 src/ | 中：需要 context 中注入 _mode |
| `templates/monorepo/java/pom.xml.jinja` | `{{ detected_java_version or java_version }}` | 低 |
| `templates/_shared/CLAUDE.md.jinja` | 构建命令用检测值 | 低 |
| `templates/app-service/` 各语言模板 | `{{ detected_java_version or java_version }}` | 低 |
| `detector.py:list_candidates()` | **回退** P1-12 spec-doc 消歧义修改 | 低：恢复原有"报告所有候选"行为 |

### 不在本次范围（后续规划）

| 项目 | 原因 |
|------|------|
| Agent 深度代码扫描 LLM 能力 | 需要 Claude Code agent 侧配合，非 init 核心职责 |
| .qoder/repowiki 知识图谱解析 | 隐藏目录内容由 Agent 消费，不做特殊解析 |
| 模板自适应生成（非填充）| 架构级变更，需独立设计 |
| run_update() 差异检测 | 已有基础实现，存量项目更新场景需单独设计 |

## 4. 验收标准

用 bug 报告中的 TMP-for-init 项目场景验证：

1. `ae init . --analyze` 检测到 `java_version=8`, `packaging=pom`, `is_multi_module=True`
2. `ae init . --incremental --defaults` 不会在根目录生成 `src/main/java/`
3. 生成的 pom.xml 中 `<java.version>` 使用检测到的版本，不是模板默认值
4. 已存在的 `tmp/pom.xml`、`tmp-manage/pom.xml` 不被覆盖
5. CLAUDE.md 包含检测到的实际命令（`mvn test` 而非 `junit`）

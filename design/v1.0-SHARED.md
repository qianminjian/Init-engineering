## 一、设计目标

### 1.1 为什么是现在这个架构

经过 v0.6.0（Bash 编排）→ v2.0（纯 Prompt）→ v2.1（Prompt + Bash 闸门）三版迭代，已经验证：

- **纯 Prompt 路线**：Agent 可能跳过 commit、跳过 Batch，确定性天花板低
- **纯 Bash 路线**：控制流确定但人机衔接断裂，Agent 能力无法充分释放
- **Claude Code 内部**：无法实现 Python 控制流（进程隔离）

三个业界框架的本质启示：**控制流在 LLM 外部，LLM 只负责各步骤内的智能工作**。

本方案采用 **Python 应用 + Claude API** 架构：
- Python 负责控制流、状态管理、Checkpoint、工具执行（确定性）
- Claude API 负责代码设计、实现、审查（智能）
- **从 Claude Code Skill 升级为独立 CLI 应用**

### 1.2 架构决策

| # | 决策 | 理由 | 日期 |
|---|------|------|------|
| 1 | Python 应用（非 Claude Code Skill） | 控制流必须在代码中，不能依赖 LLM 文本理解 | 2026-06-23 |
| 2 | Claude API（Anthropic SDK）— 方式 A | 纯 API 调用，Agent 可替换，控制流完全在 Python | 2026-06-23 |
| 3 | 先单 Agent 后多 Agent | 单 Agent 验证核心循环，多 Agent 在已验证基础上增量 | 2026-06-23 |
| 4 | 设计先出，开发后行 | 方案确认后再编码 | 2026-06-23 |

### 1.3 核心能力

| 能力 | 说明 |
|------|------|
| `ae init` | 项目环境初始化（替代 /init 工程化） |
| `ae dev-loop` | 单需求开发循环（Architect → Developer → Critic） |
| `ae dev-loop --multi` | 多 Agent 并行开发（未来迭代） |
| 确定性 Guardrail | Python 代码检查，不依赖 Agent 自觉 |
| Checkpoint/恢复 | SQLite 自动持久化，中断可恢复 |
| 质量门禁 | Lefthook + CI 生成，开箱即用 |

```python
# Auto-engineering engine/loop.py

class DevLoop:
    def tick(self) -> bool:
        """执行单次迭代，返回 True 继续，False 结束。"""
        if self.step > self.max_steps:
            self.status = "out_of_steps"
            return False

        self.current_task = self.graph.prepare_next_stage(
            self.checkpoint.state,
            self.checkpoint.pending_writes,
        )

        if not self.current_task:
            self.status = "done"
            return False

        if self.guardrail_chain.should_interrupt(self.checkpoint, self.current_task):
            self.status = "interrupt"
            return False

        return True

    def after_tick(self) -> None:
        """保存 checkpoint，更新状态。"""
        self.checkpoint.apply_writes(self.current_task.writes)
        self.checkpoint.save()
```

```python
# 参考 AutoGen _single_threaded_agent_runtime.py:466, 671
# Auto-engineering runtime/runtime.py

class AgentRuntime:
    def __init__(self):
        self._message_queue: asyncio.Queue[TaskEnvelope] = asyncio.Queue()
        self._agents: dict[str, BaseAgent] = {}
        self._subscriptions: dict[str, set[str]] = {}

    async def _dispatch_task(self, envelope: TaskEnvelope) -> None:
        """投递任务到目标 Agent，收集响应。"""
        agent = self._agents[envelope.recipient]
        response = await agent.execute(envelope.task, ctx=envelope.context)
        if response:
            self._message_queue.put_nowait(response)

    async def run_loop(self, initial_task: TaskEnvelope) -> None:
        """主消息循环。"""
        await self._message_queue.put(initial_task)
        while True:
            try:
                envelope = await self._message_queue.get()
                await self._dispatch_task(envelope)
            except QueueShutDown:
                break
```

```python
# Auto-engineering crew/process.py

class SequentialProcess:
    def execute(self, tasks: list[Task], checkpoint: Checkpoint | None = None) -> CrewOutput:
        start_idx = self._get_resume_index(tasks, checkpoint) or 0

        for i, task in enumerate(tasks[start_idx:], start=start_idx):
            agent = task.agent

            # 收集前置任务输出作为上下文
            context = self._build_context(task, tasks[:i])

            # Agent 执行：LLM 调用 + 工具调用
            output = agent.execute_task(
                task=task.description,
                expected_output=task.expected_output,
                context=context,
                tools=task.tools,
            )

            # Guardrail 验证
            self.guardrail_chain.on_execute_end(task, output)

            # 保存 checkpoint
            checkpoint.save(task_id=task.id, output=output)

        return CrewOutput.from_tasks(tasks)
```

### init 与 dev-loop 的共享契约

`ae init` 和 `ae dev-loop` 是两条独立命令，但 dev-loop 运行时需要知道项目的工程环境信息。这些信息由 init 写入 `.ae-answers.yml`，dev-loop 消费。

#### 设计原则

```
ae init 创建了项目 → .ae-answers.yml 是权威来源
项目不是 ae init 创建的 → 运行时自检测生成 .ae-answers.yml
代码与 .ae-answers.yml 不一致 → 以代码为准，分层处理
```

#### 信息分类

| 类型 | 字段示例 | 检测方式 | 不一致时 |
|------|---------|---------|---------|
| **可客观判定** | `package_manager` | 检查 lock 文件 | 以代码为准，静默更新 |
| | `use_typescript` | `tsconfig.json` 是否存在 | 同上 |
| | `ci_platform` | `.github/workflows/` / `.gitlab-ci.yml` | 同上 |
| | `test_runner` | 测试框架配置文件 | 同上 |
| | `has_lefthook` | `lefthook.yml` 是否存在 | 同上 |
| **不可客观判定** | `project_name` | — | 代码无法判断，不改 |
| | `project_description` | — | 同上 |
| | `project_type` (8 种) | 启发式签名匹配，可能模糊 | 有冲突时提示用户，不改 |

#### 解析流程

```
┌─────────────────────────────────────────────────┐
│ ae dev-loop 启动                                 │
│   │                                              │
│   ├─ 读 project_root/.ae-answers.yml             │
│   │                                              │
│   ├─ 文件存在?                                    │
│   │   ├─ 否 → 全量自检测 → 写入 .ae-answers.yml   │
│   │   │         (首次初始化，无 init 的项目也能用)  │
│   │   │                                          │
│   │   └─ 是 → 逐字段自检测                        │
│   │           ├─ 可判定项不一致 → 以代码为准       │

│   │           │                                  │
│   │           ├─ 不可判定项不一致 → 提示用户       │
│   │           │    "记录的 project_type 是 X，     │
```

```python
# crew/process.py — 当前
class SequentialProcess:
    def execute(self, tasks, checkpoint): ...

# crew/process.py — 未来
class ParallelProcess:
    """借鉴 CrewAI hierarchical + LangGraph fan-out"""
    def execute(self, tasks, checkpoint):
        dispatch_table = self._build_dispatch_table(tasks)
        results = asyncio.gather(*[
            agent.execute(task, ctx)
            for task, agent, ctx in dispatch_table
        ])

# runtime/runtime.py — 当前
class AgentRuntime:
    """单线程 Agent 运行时"""
    _task_queue: asyncio.Queue[TaskEnvelope]

# runtime/runtime.py — 未来
class MultiAgentRuntime(AgentRuntime):
    """多 Agent 并行运行时"""
    _agent_pool: dict[str, asyncio.Task]
    _contract_registry: ContractRegistry

# contracts/contract.py — 未来
class Contract:
    """借鉴 AutoGen Topic + CrewAI context passing"""
    name: str
    schema: dict   # OpenAPI fragment
    producer: AgentId
    consumers: list[AgentId]
```

## 九、关键设计决策

| # | 决策 | 理由 |
|---|------|------|
| 1 | Python + Claude API（非 Claude Code Skill） | 控制流必须在 Python 代码中，不能依赖 Agent 自觉 |
| 2 | 利用 Anthropic SDK Tool Use（非自定义协议） | 复用成熟协议，减少自有代码 |
| 3 | SQLite checkpoint（非 JSON 文件） | 事务保证一致性，支持并发读 |
| 4 | 先单 Agent 后多 Agent | 单 Agent 验证核心循环正确，多 Agent 在已验证的基础上增量 |
| 5 | Click CLI（非 Typer） | Click 是 Python CLI 事实标准，文档最全 |
| 6 | 模板采用 Copier 式 `ae-template.yml` 描述文件 | YAML 描述交互/校验/钩子，而非裸模板目录；5 个业界工具的共同选择 |
| 7 | Guardrail 是 Python 对象（非 Bash 脚本） | Python 可以和 checkpoint、state 直接交互，比 Bash 灵活 |
| 8 | init 模板采用 Copier 式 `ae-template.yml` 描述文件 | YAML 描述交互/校验/钩子，而非裸模板目录 | 2026-06-23 |
| 9 | 模板在临时目录渲染 + 原子移动 | 防止生成失败时残留文件在目标目录（Cookiecutter/Copier 已验证模式） | 2026-06-23 |
| 10 | init 不支持模板更新/迁移（v1.0 不做） | Copier update 的 3-way merge 复杂度过高，先验证核心 init 流程 | 2026-06-23 |
| 11 | `.ae-answers.yml` 为 init/dev-loop 共享契约 | dev-loop 优先读 answers 文件，fallback 自检测；可判定项以代码为准静默更新，不可判定项提示不改 | 2026-06-23 |

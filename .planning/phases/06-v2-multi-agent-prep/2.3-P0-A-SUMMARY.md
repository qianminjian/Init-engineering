# Phase 06 Plan 2.3-P0-A Summary: v2.0 Channel 体系归属决策 + LoopState 重命名

## Goal

明确 v2.0 Channel 体系是 (a) 真用 还是 (b) checkpoint 专用. 选一个不动, 消除 type 双义.

## Decision (b) — Channel 仅供 checkpoint 专用

**关键发现 (审计 Phase 1 实际代码):**
- `engine.state.LoopState` (v1.0 dataclass, 68 行) — 被 13+ 生产文件 (gates / runtime / engine) 使用, 真正的运行时共享状态
- `loop.state.LoopState` (v2.0 Pydantic, 687 行) — 真实用户只有 `checkpoint/migrate.py` + `loop/checkpoint.py` + 测试
- v2.0 Orchestrator 实际使用的是 `engine.state.LoopState` (v1.0), 不是 v2.0 Pydantic
- 双 type 双义: `LoopState` (engine v1.0) vs `LoopState` (loop v2.0) 同名两义

**Why (b):** (a) 风险大 (跨 13+ 文件, 破坏 v1.0 契约); (b) 承认 v2.0 "Channel 体系"在运行时是装饰品, 实质是 v1.0 套 Channel 名字. 重命名即可消除双义, 不强行补 v2.0 设计时未实现的"runtime state"部分.

## Why 借鉴 LangGraph 不完整实现

v2.0 设计时借鉴 LangGraph `Pregel.State` (Channel 容器同时是 checkpoint envelope 和 runtime state). 实际实现只做了一半 — 只用了 envelope 角色, runtime 还是 v1.0 dataclass.

**决策 23 选择不强行补另一半**, 因为:
1. 改 13+ 文件破坏 v1.0 契约, 风险高
2. 现有 v1.0 dataclass 设计已稳定 (BEACON 决策 1-7, 决策 11-12 明确 v2.0 是叠加)
3. 重命名 (零行为变更) 已解决概念混淆 — 后续读代码的人看到 `CheckpointEnvelope` 就知道这是 checkpoint 专用, 不会误用

借鉴 LangGraph 完整实现 (envelope + runtime) 留给未来 v3.0 评估.

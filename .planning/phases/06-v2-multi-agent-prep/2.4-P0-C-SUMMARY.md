# Phase v2.4 Plan P0-C: Coverage Gate 冻结 Summary

**CoverageGate 冻结: 决策 (b) 而非 (a), 选复杂度低/影响小路径, 真实覆盖率检查移交给 CI 独立 job.**

## 执行决策

### 决策上下文
审计 Phase 1 发现: `gates/coverage.py` 实际永远返回 "skip: 未提取到覆盖率数据" 因为本项目未装 pytest-cov, pyproject.toml addopts 不含 `--cov`. 这是 "覆盖率 gate" 但永远不工作 — 名存实亡.

### 二选一分析

| 选项 | 做法 | 风险 | 收益 |
|------|------|------|------|
| (a) 激活 | `pip install pytest-cov` + pyproject.toml addopts 加 `--cov=auto_engineering` | pytest + cov instrumentation ~2x 内存, 全量跑可能爆 16G 物理内存 (本项目硬约束) | dev-loop 内能跑出真实覆盖率 |
| (b) 冻结 | 文件头加 ⚠️ 标记 + DeprecationWarning + BEACON 决策 25 | dev-loop 内此 Gate 不产生有意义数据 | 复杂度低, 不破坏 16G 内存约束, 真实数据走 CI |

**选择 (b) 冻结**:
1. 内存约束是项目级硬禁令 (`.claude/rules/pytest-memory-management.md` 是事故复盘产物), 优先级高于 Gate 真实工作
2. 真实覆盖率检查在 CI 跑更合理 (独立环境/独立内存/独立 timing), dev-loop 单进程跑 coverage 价值有限
3. 与 builtin.py 决策 22 / engine/checkpoint.py 决策 24 同模式 — 项目已建立"冻结 + 文档"的标准流程
4. 测试只需保留 `verdict.passed` 接口 — 0 回归风险

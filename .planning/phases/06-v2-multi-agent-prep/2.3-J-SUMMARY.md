# Phase 2.3 Plan J: 内置 LLM SemanticEvaluator (P1.6, FINAL) Summary

## One-liner

内置 ClaudeSemanticEvaluator (接 AnthropicProvider) + OrchestratorConfig 默认启用 — 解除 P1.6 阻断, 让 LLM 语义评估开箱即用, 借鉴 LangGraph ConditionalEdge 模式。

## What was built

1. **`auto_engineering/loop/semantic_evaluator.py`** — 新建 150+ 行文件:
   - `ClaudeSemanticEvaluator` dataclass, 实现 `async (round_result: RoundResult) -> bool` 协议
   - 接 `AnthropicProvider` 调 Claude API (用 `asyncio.to_thread` 包装同步调用)
   - 默认 model: `claude-haiku-4-5`, max_tokens=256, temperature=0.0 (确定性评估)
   - Graceful degradation: 无 API key → True (不阻止, 让其他 Gate 决定)
   - Conservative failure: API 异常 / JSON 解析失败 → False (避免假阳性停止)
   - Prompt 模板: requirement / summary / gate_results → JSON `{satisfied, reason}`

2. **`auto_engineering/loop/orchestrator.py` OrchestratorConfig** — 加 `__post_init__`:
   - 行为契约: semantic_evaluator 已显式传入 (非 None) → 不覆盖
   - semantic_evaluator 为 None + 有 `ANTHROPIC_API_KEY` → 自动启用 `ClaudeSemanticEvaluator`
   - semantic_evaluator 为 None + 无 API key → 保持 None (Orchestrator.run() 跳过语义评估)
   - 借鉴 LangGraph ConditionalEdge: LLM 评估路由开箱即用

3. **`tests/test_loop_semantic_evaluator.py`** — 9 个新用例 (TDD RED→GREEN):
   - A. 无 API key 行为 (2 用例): 默认 True / 空 api_key 回退环境变量
   - B. mock Claude API 返回有效 JSON (2 用例): satisfied=true / satisfied=false
   - C. mock Claude API 返回 invalid JSON (2 用例): 纯文本 / 缺字段
   - D. OrchestratorConfig 默认行为 (3 用例): 有 API key 启用 / 无 API key None / 显式值不覆盖

4. **`design/BEACON.md`** — 决策 20 + 演进日志更新 + 阶段标题改 v2.3 Wave 2 完成

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Module-level AnthropicProvider import for mock patching**

- **Found during:** GREEN phase (test execution)
- **Issue:** Test 用 `patch("auto_engineering.loop.semantic_evaluator.AnthropicProvider")` mock 类, 但实现用延迟 import (`from auto_engineering.llm.anthropic_provider import AnthropicProvider` 在 `_call_claude` 函数内), mock patch 找不到目标
- **Fix:** 把 import 移到模块顶部 — 接受一次性 import 代价换取 mock-friendly 接口 (同时简化代码)
- **Files modified:** `auto_engineering/loop/semantic_evaluator.py`
- **Commit:** `f20fa37`

**2. [Rule 3 - Blocking] Test mocked create_message as AsyncMock but implementation wraps sync call in executor**

- **Found during:** GREEN phase (test execution)
- **Issue:** Test 用 `AsyncMock(return_value=...)` 但实现用 `loop.run_in_executor` 调用 sync `create_message`, 同步 mock 返回的对象被包成 coroutine 而非真实 response
- **Fix:** 改用 `MagicMock(return_value=...)` (同步 mock) — 真实场景下 `AnthropicProvider.create_message` 是同步方法
- **Files modified:** `tests/test_loop_semantic_evaluator.py`
- **Commit:** `f20fa37`

### Refactor (post-GREEN cleanup)

**3. [Improvement] asyncio.to_thread 替代手写 _run_in_executor**

- **Found during:** REFACTOR phase
- **Issue:** 之前的 `_run_in_executor` helper 是 `run_in_executor` + `functools.partial` 包装, Python 3.9+ 有更简洁的 `asyncio.to_thread`
- **Fix:** 删除 `_run_in_executor` helper, 用 `asyncio.to_thread(provider.create_message, ...)` 替代, 顶部 import asyncio
- **Files modified:** `auto_engineering/loop/semantic_evaluator.py`
- **Commit:** `f60c209`
